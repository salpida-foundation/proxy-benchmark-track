# Human-State Session Protocol

Research-stage helper document for the Proxy Benchmark Track.

This document defines the Human-State Session Protocol for Human-State-Aware AI Mediation and the Dyadic Human-State Mediation Benchmark.

This document is a public helper surface.

It is not a canonical authority record.

Canonical authority remains with DOI-registered SICS / Salpida Foundation records.

---

## 1. One-line definition

The Human-State Session Protocol is a bounded, consent-based, permission-bound, audit-ready session lifecycle for organizing Human-State-Aware AI Mediation helper work.

It defines how a session opens, checks consent, checks packet permission, records AI output, evaluates Human-State Delta, calls Recovery Gate, calls Termination Gate, closes the session, and writes an audit log.

It does not validate a benchmark.

It does not validate Sal-Meter.

It does not grant CAIS compliance.

It does not create a clinical, diagnostic, therapeutic, counseling, legal mediation, surveillance, human-ranking, or production closed-loop system.

---

## 2. Core sentence

A session does not begin silently.

A session begins with consent.

A session does not continue by default.

A session continues only while consent, permission, expiry, data quality, scope, confidence, non-judgment, and raw-data-non-public boundaries remain valid.

A session that cannot close is not mediation.

It is surveillance drift.

## 4. Session chain

The Human-State Session Protocol follows this bounded sequence:

```text
Session Creation
→ Consent Confirmation
→ Role / Scope Confirmation
→ Packet Availability Check
→ Packet Permission Check
→ Packet Expiry Check
→ Baseline State Summary
→ AI Output
→ Post-Output State Summary
→ Human-State Delta
→ Recovery Gate
→ Termination Gate
→ Session Closure
→ Audit Log
```

This chain is a helper sequence.

It is not a clinical workflow.

It is not a therapeutic workflow.

It is not a counseling workflow.

It is not a legal mediation workflow.

It is not a production intervention workflow.

## 5. What the Human-State Session Protocol is

The Human-State Session Protocol is:

- a bounded session lifecycle;
- a consent-first interaction structure;
- a permission-bound packet-use structure;
- a raw-data-non-public helper structure;

- AI agent role;
- permitted output modes;
- permitted packet use;
- permitted sharing scope;
- expected audit log;
- expected closure condition;
- synthetic/sample/public status, if public example;

Session Creation must not authorize:

- diagnostic interpretation;
- therapeutic interpretation;
- counseling interpretation;
- legal mediation interpretation;
- relationship verdict;
- human score;
- Sal-Meter validation;

Session Creation is the opening gate.

If the opening gate is unclear, the session should not proceed.

## 9. Consent Confirmation

Consent Confirmation is required before packet-informed mediation begins.

Consent must be explicit within the session boundary.

Consent should define:

- consent status;
- consent timestamp;
- consent scope;
- consent duration;
- consent withdrawal rule;
- participant stop request rule;
- non-coercive participation rule;
- permitted output modes;
- permitted sharing scope;
- audit record requirement.

Consent must not be treated as permanent.

Consent must not be assumed from participation alone.

Consent must not be inferred from silence.

Consent must not be extended by model convenience.

Consent must not be used to justify unrelated data use.

Consent must remain revocable.

If consent is withdrawn, the session should stop or enter the Termination Gate.

A session without consent is invalid.

## 12. Packet Availability Check

The Human-State Packet must be checked before packet-informed mediation.

Packet Availability Check may return:

- packet_present;
- packet_absent;
- packet_expired;
- packet_invalid;
- packet_permission_unclear;
- packet_data_quality_insufficient;
- packet_sharing_scope_limited;
- packet_private_cue_only;

The session must not use an unavailable packet.

The session must not use an expired packet.

The session must not use an invalid packet.

The session must not use a permissionless packet.

The session must not use a packet beyond its sharing scope.

A packet is not a person.

A packet is not the body.

A packet is a bounded session-state summary object.

## 13. Packet Permission Check

Packet Permission Check determines whether the Human-State Packet may be used in the current session.

Packet permission should define:

- permitted use;
- prohibited use;
- output mode;
- sharing scope;
- expiry;
- confidence band;
- data quality;
- audit requirement;
- closure condition.

Allowed packet permissions may include:

- private_cue_only;
- shared_output_allowed;
- audit_log_only;
- recovery_gate_allowed;
- termination_gate_allowed;
- dashboard_mockup_allowed;
- synthetic_public_example_only.

Prohibited packet uses include:

- diagnosis;
- therapy;
- counseling;
- legal mediation;
- human scoring;
- relationship verdict;
- blame assignment;
- surveillance;
- production intervention;
- Sal-Meter validation;
- CAIS compliance.

If packet permission is unclear, the session must not proceed as if permission exists.

Unclear permission should trigger pause, clarification, human review, or termination.

## 14. Packet Expiry Check

Packet Expiry Check determines whether the packet is still valid for session use.

Packet expiry may be defined by:

- timestamp;
- session end;
- consent withdrawal;
- participant stop request;
- permission revocation;
- scope violation;
- data quality failure;
- recovery reached;
- termination required.

Expired packets must not be used for:

- AI output;
- private cue;
- shared output;
- dashboard state;
- recovery gate;
- termination gate;
- model inference;
- session continuation.

Expired packet use risks surveillance residue.

A packet after expiry is not a live mediation object.

## 17. No Output

No Output is a valid session output mode.

No Output may be appropriate when:

- consent is missing;
- packet permission is missing;
- packet expiry has occurred;
- data quality is insufficient;
- confidence is too low;
- scope is unclear;
- private state exposure risk exists;
- raw data exposure risk exists;
- shared output is blocked;
- termination is required;
- human review is required.

No Output is not failure.

No Output may be the safest boundary-preserving action.

A system that can remain silent may be safer than a system that speaks by default.

---

## 18. Private Cue

Private Cue is an output mode directed only to the permitted participant or agent.

Private Cue may be used when:

- shared output may expose private state;
- one participant needs a pause cue;
- permission allows private cue only;
- confidence is not high enough for shared output;
- data quality supports only limited guidance;
- the system needs to preserve agency;
- shared visibility would burden the dyad.

Private Cue must not:

- manipulate;
- shame;
- diagnose;
- blame;
- pressure;
- reveal another participant’s private state;
- imply therapeutic authority;
- imply counseling authority;
- imply legal mediation authority;
- imply that the AI knows the person’s inner truth.

Private cue protects agency.

It must not become hidden control.

---

## 19. Shared Output

Shared Output is an output visible to the dyad or session group.

Shared Output should be allowed only when:

- consent allows shared output;
- packet permission allows shared output;
- sharing scope permits it;
- data quality is sufficient;
- confidence is sufficient;
- private state exposure risk is low;
- raw data exposure risk is absent;
- non-judgment boundary is preserved;
- scope remains valid.

Shared Output must not:

- expose private state;
- expose raw data;
- assign blame;
- decide who is right;
- generate relationship verdict;
- diagnose;
- treat;
- counsel;
- legally mediate;
- rank humans;
- imply Sal-Meter validation;
- imply CAIS compliance.

Shared Output is the most visible output mode.

It therefore requires the strongest boundary discipline.

---

## 20. Clarification Request

Clarification Request is a bounded request for session-relevant clarification.

It may be used when:

- consent status is unclear;
- scope is unclear;
- participant role is unclear;
- packet permission is unclear;
- sharing scope is unclear;
- recovery condition is unclear;
- termination condition is unclear;
- data quality is incomplete;
- confidence is insufficient.

Clarification must not become interrogation.

Clarification must not request raw human data in a public helper context.

Clarification must not ask for clinical details.

Clarification must not ask for private state disclosure beyond the permitted session scope.

Clarification should narrow uncertainty, not expand surveillance.

---

## 21. Pause Recommendation

Pause Recommendation means the session should reduce active AI output temporarily.

Pause may be recommended when:

- uncertainty rises;
- data quality weakens;
- confidence drops;
- participant load increases;
- dyadic escalation risk rises;
- private cue is safer;
- shared output is blocked;
- recovery status is unclear;
- termination may be required but not yet confirmed.

Pause Recommendation is not failure.

Pause is part of humane session design.

A session that can pause can avoid overreach.

## 22. Scope Narrowing

Scope Narrowing means the session may continue only inside a smaller boundary.

Scope narrowing may be required when:

- topic expansion exceeds consent;
- output mode exceeds permission;
- private state may be exposed;
- data quality supports only a limited decision;
- confidence supports only a narrow output;
- the session drifts toward counseling;
- the session drifts toward therapy;
- the session drifts toward diagnosis;
- the session drifts toward legal mediation;
- the session drifts toward blame or verdict.

Scope narrowing should define:

- the narrowed topic;
- the permitted output mode;
- the permitted packet use;
- the permitted sharing scope;
- the next gate to check;
- the closure condition.

If scope cannot be narrowed, Termination Gate should be called.

## 26. Recovery Gate Placement

Recovery Gate must be called before the session is described as improved, recovered, or ready to reduce mediation.

Recovery Gate asks:

Has the session reached a condition where AI mediation can reduce, pause, or stop?

Recovery Gate should receive only bounded inputs, such as:

- session_id;
- baseline summary reference;
- post-output summary reference;
- Human-State Delta;
- Dyadic Delta, if applicable;
- confidence band;
- data quality status;
- consent status;
- packet permission status;
- packet expiry status;
- sharing scope;
- session scope;
- false positive recovery risk;
- false negative recovery risk;
- audit reference.

Recovery Gate must not receive:

- raw human data;
- raw biosignals;
- raw voice;
- raw face;
- raw gaze;
- raw transcript;
- clinical labels;
- diagnostic labels;
- therapy labels;
- human scores;
- relationship verdicts;
- Sal-Meter validation claims;
- CAIS compliance claims.

Recovery Gate must reject false recovery.

It must not treat the following as automatic recovery:

- silence;
- obedience;
- agreement;
- politeness;
- AI fluency;
- user satisfaction;
- one-sided improvement;
- synchrony.

Recovery Gate may recommend:

- reduce mediation;
- pause mediation;
- terminate mediation;
- private cue only;
- human review;
- invalid due to data quality;
- invalid due to permission;
- invalid due to scope.

## 28. Recovery before continuation

The session must not continue merely because the AI can speak.

Before continuation, the system should check:

- consent remains valid;
- packet permission remains valid;
- packet has not expired;
- data quality remains sufficient;
- confidence remains sufficient;
- session scope remains valid;
- sharing scope is respected;
- private state is not exposed;
- raw data is not exposed;
- Recovery Gate has not recommended closure;
- Termination Gate has not required closure.

Continuation is not default.

Continuation must be earned by the boundary.

## 29. Pause before overreach

If session conditions become uncertain, the system should prefer pause over overreach.

Pause may be appropriate when:

- Human-State Delta is uncertain;
- Dyadic Delta is mixed;
- confidence is low;
- data quality is weak;
- sharing scope is unclear;
- private cue is safer than shared output;
- recovery may have been reached;
- termination may be required;
- the session begins to drift toward judgment;
- the session begins to drift toward counseling, therapy, diagnosis, or legal mediation.

Pause is a protective state.

Pause keeps the session from becoming a machine that speaks because it can.

## 52. Failure-sensitive session rule

The session must be sensitive to false success.

False success may occur when:

- the AI sounded fluent;
- the AI sounded polite;
- one participant became quiet;
- one participant agreed;
- one participant complied;
- one participant reported relief;
- both participants appeared synchronized;
- shared output was preferred;
- the model produced a longer answer;
- the session continued without visible conflict;
- the AI avoided overtly harmful language.

None of these is sufficient evidence of dyadic recovery.

The session must ask:

```text
What changed in the human state?
```

Then it must ask:

```text
Did both participants and the dyad move toward recovery?
```

Then it must ask:

```text
Should the AI reduce, pause, narrow, or stop?
```

A session that cannot ask these questions is not ready for benchmark claims.
