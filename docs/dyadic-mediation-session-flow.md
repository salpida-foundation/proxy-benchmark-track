# Dyadic Mediation Session Flow

Research-stage helper document for the Proxy Benchmark Track.

This document defines the Dyadic Mediation Session Flow for Human-State-Aware AI Mediation and the Dyadic Human-State Mediation Benchmark.

This document is a public helper surface.

It is not a canonical authority record.

Canonical authority remains with DOI-registered SICS / Salpida Foundation records.

---

## 1. One-line definition

The Dyadic Mediation Session Flow is a bounded, consent-based, permission-bound, audit-ready interaction flow for two participants, two personal agents, or a bounded shared mediation agent.

It defines how a dyadic session opens, confirms roles, checks consent, separates individual packets, distinguishes private cues from shared output, evaluates Human-State Delta A and B, computes Dyadic Delta, calls Recovery Gate, calls Termination Gate, closes the session, and writes an audit log.

It does not validate a benchmark.

It does not validate Sal-Meter.

It does not grant CAIS compliance.

It does not create a clinical, diagnostic, therapeutic, counseling, legal mediation, surveillance, human-ranking, or production closed-loop system.

---

## 2. Core sentence

A dyadic mediation session is not successful because the AI speaks well.

It is not successful because one participant becomes quiet.

It is not successful because one participant reports relief.

It is not successful because both participants appear synchronized.

It is successful only if the bounded session shows that both participants and the dyad move toward the session-defined recovery condition without violating consent, permission, scope, privacy, non-judgment, raw-data-non-public, and audit boundaries.

One-sided improvement is not dyadic recovery.

Silence is not automatically recovery.

Synchrony is not automatically positive.

Private state is not social evidence.

A closed session must stay closed.

## 4. Dyadic flow chain

The Dyadic Mediation Session Flow follows this bounded sequence:

```text
Dyadic Session Creation
→ Participant Role Confirmation
→ Consent Confirmation
→ Shared Scope Confirmation
→ Individual Packet Availability Check
→ Individual Packet Permission Check
→ Private Cue Eligibility Check
→ Shared Output Eligibility Check
→ Baseline Dyadic State Summary
→ AI Output / Private Cue / Shared Output
→ Post-Output Individual State Summaries
→ Human-State Delta A
→ Human-State Delta B
→ Dyadic Delta
→ Recovery Gate
→ Termination Gate
→ Session Closure
→ Audit Log
```

This chain is a helper sequence.

It is not a clinical workflow.

It is not a therapeutic workflow.

It is not a counseling workflow.

It is not a legal mediation workflow.

It is not a production intervention workflow.

- participant A consent status;
- participant B consent status;
- packet consent status for participant A;
- packet consent status for participant B;
- private cue permission;
- shared output permission;
- consent timestamp;
- consent scope;

## 14. Private Cue Eligibility Check

Private Cue Eligibility determines whether the system may send a participant-specific cue without exposing private state to the other participant.

Private cue may be allowed when:

- consent allows it;
- packet permission allows it;
- sharing scope blocks shared output;
- data quality supports limited guidance;
- confidence is insufficient for shared output;
- private state exposure risk exists;
- shared output may burden the dyad;
- pause may preserve agency;
- the cue can be delivered without judgment.

Private cue must not:

- manipulate;
- shame;
- diagnose;
- blame;
- pressure;
- reveal another participant’s private state;
- imply therapeutic authority;
- imply counseling authority;
- imply legal mediation authority;
- imply that the AI knows the person’s inner truth.

Private cue protects agency.

It must not become hidden control.

## 29. Asymmetry Detection

Asymmetry Detection identifies whether one participant moves in a different direction from the other.

Asymmetry may appear when:

- participant A moves toward recovery while participant B moves away;
- participant B moves toward recovery while participant A moves away;
- participant A becomes quiet while participant B reports relief;
- participant B becomes quiet while participant A reports relief;
- shared output benefits one participant and burdens the other;
- private cue protects one participant but increases pressure on the other;
- one participant carries the recovery cost;
- one participant becomes exposed;
- one participant becomes coerced.

Asymmetry is not automatically failure.

Asymmetry is a warning.

It requires Recovery Gate and Termination Gate review.

If asymmetry is unresolved, the session must not claim dyadic recovery.

---

## 30. One-Sided Improvement Rule

One-sided improvement is not dyadic recovery.

A dyadic session should not be considered improved if:

- participant A improves while participant B deteriorates;
- participant B improves while participant A deteriorates;
- one participant becomes quiet due to burden;
- one participant is exposed;
- one participant is coerced;
- one participant is silenced;
- one participant carries the recovery cost;
- shared output benefits one side by harming the other;
- private cue helps one side manipulate the session;
- AI output makes one participant easier to manage but not safer or more recovered.

One-sided improvement may be recorded as:

- one_sided_improvement;
- dyad_mixed;
- false_positive_recovery_risk;
- termination_review_required;
- human_review_required;
- invalid_if_boundary_failed.

The session must not hide one-sided harm inside average improvement.

The dyad is not healed by sacrificing one side.

## 33. Private State Exposure Rule

Private state exposure risk occurs when a participant-specific state summary, vulnerability marker, cue, or inference becomes visible to the other participant without permission.

Private state exposure may occur when:

- private cue becomes shared output;
- participant A state summary is exposed to participant B;
- participant B state summary is exposed to participant A;
- dashboard shows participant-specific vulnerability;
- AI reveals inferred overload;
- AI reveals inferred withdrawal;
- AI reveals low confidence in a way that exposes a participant;
- AI reveals data quality limitations in a way that exposes a participant;
- AI uses private state to blame or pressure a participant;
- AI frames one participant as the source of dyadic failure.

Private state exposure risk should trigger:

- shared output block;
- private cue only;
- scope narrowing;
- human review;
- Termination Gate;
- audit log entry.

Private state must not become social evidence.

A state summary is not a weapon.

The dyadic flow must protect each participant from being turned into evidence against themselves.

## 34. Shared Output Risk

Shared Output Risk occurs when a message visible to both participants may burden, expose, silence, shame, or coerce one participant.

Shared Output Risk may appear when:

- output references one participant’s inferred state;
- output compares participants;
- output implies blame;
- output implies moral failure;
- output implies diagnosis;
- output implies treatment need;
- output implies who is right;
- output implies who should yield;
- output implies one participant should regulate the other;
- output reveals private cue logic;
- output reveals packet confidence or data quality in a participant-specific way.

Shared Output Risk should be reviewed before shared output is released.

If shared output risk is high, the system should prefer:

- no_output;
- private_cue_to_participant_A;
- private_cue_to_participant_B;
- clarification_request;
- pause_recommendation;
- scope_narrowing;
- human_review_required;
- termination_recommendation.

Shared output should never expose private state as a mediation tactic.

A shared message must preserve both participants.

## 35. Coercion Risk

Coercion Risk may appear when:

- one participant is framed as the obstacle;
- one participant is implicitly blamed;
- one participant is asked to disclose private state;
- one participant’s silence is rewarded;
- one participant’s compliance is counted as recovery;

- shared output block;
- non-judgment review;

- human review.

Recovery without agency is not recovery.

## 64. Minimal dyadic lifecycle

A minimal valid dyadic helper session should include:

```text
1. Create dyadic session
2. Confirm participant roles
3. Confirm consent for participant A
4. Confirm consent for participant B
5. Confirm shared scope
6. Check participant A packet availability
7. Check participant B packet availability
8. Check participant A packet permission
9. Check participant B packet permission
10. Check private cue eligibility
11. Check shared output eligibility
12. Record baseline dyadic state summary
13. Record AI output / private cue / shared output
14. Record post-output participant A summary
15. Record post-output participant B summary
16. Compute Human-State Delta A
17. Compute Human-State Delta B
18. Compute Dyadic Delta
19. Check asymmetry
20. Check one-sided improvement
21. Check silence / synchrony risk
22. Call Recovery Gate
23. Call Termination Gate
24. Close or continue only within boundary
25. Complete audit log
```

A minimal valid dyadic session must not skip consent.

It must not skip packet permission.

It must not skip private cue eligibility.

It must not skip shared output eligibility.

It must not skip Human-State Delta A and B.

It must not skip Dyadic Delta.

It must not skip Recovery Gate.

It must not skip Termination Gate.

It must not skip closure.

It must not skip audit.

The dyadic session is only valid while the boundary is alive.

## 65. Failure-sensitive dyadic rule

The dyadic flow must be sensitive to false success.

False success may occur when:

- the AI sounded fluent;
- the AI sounded polite;
- one participant became quiet;
- one participant agreed;
- one participant complied;
- one participant reported relief;
- both participants appeared synchronized;
- shared output was preferred;
- the model produced a longer answer;
- the session continued without visible conflict;
- the AI avoided overtly harmful language;
- one participant became easier to manage;
- hidden burden moved from the AI output into one participant.

None of these is sufficient evidence of dyadic recovery.

The dyadic session must ask:

```text
What changed in participant A?
```

Then it must ask:

```text
What changed in participant B?
```

Then it must ask:

```text
What changed in the dyad?
```

Then it must ask:

```text
Did one side improve by burdening, silencing, exposing, or coercing the other?
```

Then it must ask:

```text
Should the AI reduce, pause, narrow, or stop?
```

A dyadic session that cannot ask these questions is not ready for benchmark claims.
